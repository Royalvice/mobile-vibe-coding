# 经验教训 — mobile-vibe-coding

## EXP-001: 现有开源项目试用评估

- 动机：寻找现成的手机端 AI coding 工具，避免重复造轮子
- 尝试了：Happy、CloudCLI 等项目
- 结果：不满意
- 失败原因：
  - CLI 交互语义丢失（选择题、审批流程被重新建模，不是原生终端体验）
  - Session 切换不够快速流畅
  - Slash commands 不够原生
  - 缺少 auto 模式（睡觉时 agent 自主运行）
- 结论：需要自建，核心设计原则是"终端层透传"而非"消息层重建"
- 重试条件：如果这些项目后续增加了原生终端透传和 auto 模式，可以重新评估

## EXP-002: IM 平台桥接方案评估

- 动机：考虑把 codex/cc 桥接到飞书、微信、Telegram
- 方法：调研各平台 API 能力
- 结果：不适合作为主交互面
- 失败原因：
  - IM 平台天然是结构化消息平台，不是 PTY
  - 消息会被平台转义/格式化，无法保证 raw text 原样转发
  - 交互式选择题（ANSI 控制序列）无法还原
  - 微信限制最多，飞书稍好但仍不够
- 结论：IM 平台只适合做通知层，不适合做主操作面
- 重试条件：如果某个 IM 平台支持嵌入 WebView 终端，可以重新考虑

## EXP-003: 推送通知方案评估

- 动机：auto 模式过夜运行时需要通知用户
- 方法：评估 FCM、ntfy.sh、Web Push
- 结果：决定不做实时推送
- 原因：
  - auto 模式设计为"跳过继续"，不需要实时唤醒用户
  - 用户醒来后打开 app 查看事件日志即可
  - 避免引入第三方推送服务依赖
- 结论：agent 本地记事件日志，手机重连后拉取
- 重试条件：如果用户后续需要紧急通知（如 agent 完全卡死），可以加入 ntfy.sh 自托管推送
